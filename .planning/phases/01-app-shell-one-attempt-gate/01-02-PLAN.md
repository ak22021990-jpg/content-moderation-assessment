## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-06 | Spoofing | Candidate types someone else's name/email | accept | Identity is candidate-declared per PROJECT.md; verified externally by recruiter. Not a Phase 1 threat. |
| T-01-07 | Tampering | Candidate bypasses ALREADY_COMPLETED by clearing localStorage | accept | Client-side flag is by-design defeatable; Phase 5 server-side dedup is authoritative. Already documented in Wave 1 T-01-01. |

| T-01-10 | Tampering | Browser default validation bubble competes with app validators | mitigate | noValidate on the <form> per RESEARCH.md §Pitfalls "new: forgetting to noValidate". |
| T-01-11 | Information disclosure | XSS via name/email echoed in DOM | mitigate | React text-node auto-escaping; no dangerouslySetInnerHTML in either component. |

<risks>
Wave-specific risks:

1. **Task ordering coupling.** LandingScreen imports AlreadyCompletedScreen. If Task 01-02-01 runs first, the executor must either (a) create AlreadyCompletedScreen inline temporarily and refactor in Task 01-02-02, or (b) reorder to run 01-02-02 first. Recommendation: run 01-02-02 first — smaller task, unblocks 01-02-01 cleanly.
2. **Autofocus test flakiness under happy-dom.** happy-dom's focus tracking is generally reliable but requires the element to be actually in the DOM at effect-time. Mitigation: `useEffect(() => nameRef.current?.focus(), [])` runs after mount; test asserts `document.activeElement === nameInput` after `await` on render's returned promise (Testing Library render is sync but React 19 batches effects — a single `await waitFor(...)` handles the edge case).
3. **user-event v14 async subtleties.** Every `user.type/click/tab/keyboard` call is async and MUST be awaited (RESEARCH §7 Rule 2). Missing awaits will produce test flakiness only visible in CI. Mitigation: lint pattern in code review + the tests-first RED phase catches missing awaits before commit.
4. **Enter-to-submit inside email field.** With `type="email"` on the input and a disabled submit button, some browsers still consume Enter as a form submit request. handleSubmit's early return when !canSubmit is the defense — verified by the "invalid + Enter → onStart NOT called" test.
5. **AlreadyCompletedScreen brand-safety on copy edit.** If future edits add a recruiter-contact snippet with an @company.com pattern, the copy must remain brand-neutral. Mitigation: brand-guard pre-commit hook + the textContent scan in tests.
</risks>
