          <ul>

              <li key={l2.id}>

Wave-specific risks:

1. **Fixture-vs-real taxonomy divergence.** Fixture tests use a 2-L1 mock; integration test uses real taxonomy. If the schema drifts between them, the integration test catches it. Mitigation: single integration test runs against the real JSON as a canary.
2. **vi.stubEnv API quirks in Vitest 4.x.** Vitest supports env stubbing but the exact API can vary (`vi.stubEnv`, `vi.stubGlobal`). Executor should spike the DEV=false test first; if `stubEnv` doesn't hit `import.meta.env.DEV` (which Vite processes at build time as a static replacement), fall back to a build-time constant test — assert the source code has the `import.meta.env.DEV &&` guard rather than runtime-toggling it. This meets D-P1-09 intent (guard exists) even if runtime flip is impractical.

5. **Fixture-tests bypassing GUIDE-06 spirit.** Using a fixture with `label:'CatA'` proves data-driven behavior but doesn't prove real labels render. The integration test closes that gap — do NOT delete it.
</risks>
