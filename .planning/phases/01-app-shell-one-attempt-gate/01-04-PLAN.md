App structure (JSX skeleton, prose form so executor can implement without inline code):

  1. `import { SCREENS } from './state/screens.js'`
  2. `import { useAssessmentState } from './hooks/useAssessmentState.js'`
  3. `import { useOneAttemptGuard } from './hooks/useOneAttemptGuard.js'`
  4. Import all 4 screen components + ErrorBoundary
  5. Default-exported function App():
     - `const state = useAssessmentState();`
     - `const guard = useOneAttemptGuard();`
     - Render tree:
       <ErrorBoundary>
         switch (state.screen) {
           case SCREENS.LANDING:
             → <LandingScreen onStart={state.startAssessment} hasAttempted={guard.hasAttempted} />
             (note: LandingScreen itself short-circuits to AlreadyCompletedScreen when hasAttempted=true — App does not need a separate ALREADY_COMPLETED case for the initial-mount path, BUT the SCREENS.ALREADY_COMPLETED case still exists for programmatic transitions from other screens if ever needed — render <AlreadyCompletedScreen /> for that case)
           case SCREENS.GUIDELINES:
             → <GuidelinesScreen onBegin={state.enterAssessment} />
           case SCREENS.ASSESSMENT_PLACEHOLDER:
             → <AssessmentPlaceholderScreen onReset={handleDevReset} />
             (handleDevReset = () => { guard.clear(); state.resetAttempt(); })
           case SCREENS.ALREADY_COMPLETED:
             → <AlreadyCompletedScreen />
           default:
             → <AlreadyCompletedScreen /> or throw (executor decides; recommend rendering AlreadyCompletedScreen as safe fallback + logging via console.warn)
         }
       </ErrorBoundary>

  6. NO useEffect at root; NO StrictMode wrapping added (main.jsx concern).
  7. NO new dependencies; NO CSS import beyond what Phase 0 already has (Phase 6 handles design).

State/screen invariants after composition:
  - Initial mount with clean storage: screen === LANDING (from useAssessmentState initial), guard.hasAttempted === false → LandingScreen renders form
  - Initial mount with populated sessionStorage identity: screen === GUIDELINES (IDENT-04 mid-refresh recovery)
  - Initial mount with populated cma_attempt_v1: screen === LANDING BUT guard.hasAttempted === true → LandingScreen short-circuits to AlreadyCompletedScreen
  - startAssessment → screen becomes GUIDELINES; sessionStorage written
  - enterAssessment → screen becomes ASSESSMENT_PLACEHOLDER
  - handleDevReset (only invoked from AssessmentPlaceholderScreen in DEV): both stores cleared; screen returns to LANDING

<risks>
Wave-specific risks:

1. **CI deploy failure hidden behind a passing local build.** GitHub Actions may fail on lockfile mismatch, LFS pointer regression (unlikely — no video changes in Phase 1), or a Vite-plugin-react version conflict. Mitigation: Task 01-04-02 explicitly polls CI status before advancing to the human-verify checkpoint.
2. **Human-verify checkpoint blocks on flaky Pages caching.** GitHub Pages sometimes caches the old bundle for 5-10 minutes after a deploy. Mitigation: human-verify step 1 uses an incognito window (bypasses local cache); if the old <h1> shows, the executor waits and retries before opening a repair task.
3. **StrictMode double-invocation surprises.** If main.jsx wraps in <StrictMode>, useState lazy initializers in the hooks run twice on mount (idempotent per RESEARCH §Pitfalls). Integration tests do NOT wrap in StrictMode. Should a runtime-only StrictMode bug appear, add a targeted StrictMode-wrapped test — Wave 1 T-01-04 hook tests cover the storage-read-twice case.
4. **Integration test timing under happy-dom.** The transition LANDING→GUIDELINES→PLACEHOLDER involves multiple setState calls in the same tick. userEvent.click awaits until React flushes; if a test flakes, wrap the assertion in `await waitFor(() => ...)`. Do NOT downgrade to fireEvent — RESEARCH §7 explicitly forbids it for controlled inputs.
5. **taxonomy.json schema change breaking GuidelinesScreen.** The GuidelinesScreen has optional-field pattern for definition/example (won't crash on nulls), but a structural change (e.g., renaming `subcategories` → `children`) would break rendering. Mitigation: Wave 1 taxonomy.test.js locks the structural fields; any structural change fails the schema test first.
6. **Live URL humans checkpoint dependency on network.** If GitHub Pages is down at checkpoint time, verification cannot complete. Mitigation: retry after 15-min; escalate to user if Pages remains unavailable > 1 hour.
</risks>
