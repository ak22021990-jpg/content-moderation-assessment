<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| playlist.json → <video src> | Data-driven URL injection into the video element |
| User interaction → media-chrome controls | User clicks play/seek/mute/volume — handled by media-chrome internally |
| <video> error event → error UI | Error propagation to parent component for Phase 3 handling |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Denial of service | Video src URL points to non-existent file | mitigate | Error state UI shown; onError callback exposed so parent can act (Phase 3 timer abort) |
| T-02-06 | Information disclosure | CSS reveals internal structure | accept | CSS is publicly accessible in any web app; class names are generic (cma-*) |
| T-02-07 | Elevation of privilege | canplaythrough callback exposed to parent | accept | Callback is by-design; Phase 3 timer subscribes — this is the intended contract |
| T-02-08 | Tampering | Candidate modifies video src via DevTools | accept | Client-side only; no auth/validation gate. Assessment integrity relies on Phase 5 server-side dedup, not client security. |
</threat_model>

<risks>
Wave-specific risks:

1. **media-chrome mock may diverge from real component API.** We mock PascalCase React wrappers but real components are web components. If the real API changes (e.g., a prop name) between media-chrome versions, mock tests stay green while browser breaks. Mitigation: manual browser test verification step (human-verify planned for Plan 04 wave); Playwright E2E in Phase 6.
2. **CSS --media-* property names may change in media-chrome.** Mux documents these as part of the public API. Mitigation: pin media-chrome version; if upgrading, re-audit UI-SPEC.css overrides against the new version's Custom Properties.
3. **canplaythrough fires before the component is mounted in a test context.** The useEffect cleanup handles double-fire. Mitigation: useRef guards prevent stale closure issues.
4. **Error state triggered by non-fatal events.** Some browsers fire `error` on benign network hiccups (e.g., offline→online transition). Mitigation: error state is a visual fallback, not a destructive action; page refresh recovers.
</risks>
