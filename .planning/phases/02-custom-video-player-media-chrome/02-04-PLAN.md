<risks>
Wave-specific risks:

1. **Integration test mock mismatch with real VideoPlayerScreen.** The integration test mocks VideoPlayerScreen as a simple div. If VideoPlayerScreen's real API changes (new required props), the integration test stays green while the real app breaks. Mitigation: manual browser verification (Task 02-04-02) catches this.
2. **AssessmentPlaceholderScreen.jsx referenced by undeleted file.** If another component (e.g., a test helper) imports AssessmentPlaceholderScreen, deleting the file will break the build. Mitigation: `npm run build` catches import errors; `rg` search before delete.
3. **canplaythrough fires before App.jsx mounts the onReady handler.** The `canplaythrough` event is asynchronous — the useEffect that attaches the listener runs after mount, so timing is correct. However, if the video's preload="metadata" loads fast enough and caches it, the event may fire between DOM creation and React's useEffect attachment. Mitigation: check video.readyState >= HAVE_ENOUGH_DATA in the useEffect (if already ready, call onReady synchronously).
4. **Cross-browser differences in media-chrome keyboard handling.** Safari and Firefox may have different default keyboard event behavior (e.g., Space scrolls the page in Firefox if not on a button). Mitigation: media-chrome handles preventDefault for Space internally; manual test confirms.
5. **Dev reset button visible in production.** The Phase 1 AssessmentPlaceholderScreen had a visible reset button. VideoPlayerScreen passes through onReset but should NOT render a reset button for candidates. Mitigation: code review or manual test — if a reset button exists in the player, remove it before merge.
</risks>
