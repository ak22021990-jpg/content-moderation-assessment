# Phase 2 — UI Design Contract

> Visual and interaction contract for the custom video player phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Design direction:** Cinematic streaming-service aesthetic — dark premium backgrounds, rich blue accents, clean white typography on dark surfaces. Desktop-first (moderators work on desktops per ARCHITECTURE.md out-of-scope mobile-first exclusion). Phase 6 adds GSAP transitions later; this contract defines the static visual system.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (manual CSS in `src/components/player/VideoPlayer.css`) |
| Preset | not applicable |
| Component library | none (pure CSS + custom components; media-chrome web components for player controls) |
| Icon library | none (media-chrome provides built-in control icons; use Unicode for fallback indicators) |
| Font | System font stack: `-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif` |
| CSS methodology | BEM-lite: single `.cma-*` namespace, no preprocessor. CSS custom properties for tokens. |

**No external dependencies.** Zero Google Fonts, zero icon CDNs, zero component libraries beyond `media-chrome@^4.19.2` (already the player mechanism per RESEARCH.md Section 1).

## Spacing Scale

| `--cma-space-sm` | 8px | Compact element spacing, control-bar internal padding |
| `--cma-space-md` | 16px | Default element spacing, section padding, form field gaps |
| `--cma-space-lg` | 24px | Section padding, player-to-tagpanel gap |
| `--cma-space-xl` | 32px | Layout gaps, screen-level padding |
| `--cma-space-2xl` | 48px | Major section breaks, screen heading top margin |

Exceptions:
- `--cma-player-control-gap`: `12px` — control-bar button spacing (not on 8px grid; matches media-chrome default inline spacing)
- `--cma-player-seek-thumb-active`: `4px` — seek bar thumb height expansion on hover (animation detail, not layout)

### Accent Reserved For

Accent (`#0085FF` / `#1A9AFF`) reserved exclusively for:
- Primary CTA buttons (background)
- Seek bar filled track
- Seek bar thumb
- `:focus-visible` outline ring on all interactive elements
- Chapter marker dots on seek bar (these share warning color — see Decision 1)

**NOT** used for: secondary text, headings, borders, backgrounds, icons (media-chrome icons are white by default).

### Player Container

- Max-width: `960px` (matches 16:9 at reasonable desktop size)
- Centered in viewport via `margin: 0 auto`
- `border-radius: 8px` on outer container
- `overflow: hidden` to clip media-chrome rounded corners
- `box-shadow: 0 4px 24px rgba(0, 0, 0, 0.5)` — subtle elevation over deepest background
- Background: `--cma-bg-deepest` visible as letterbox bars when video aspect ratio doesn't fill container

### Video Title Bar

- Background: `--cma-bg-surface` (`#141725`)
- Padding: `12px 16px` (custom, tighter than spacing scale for chrome density)
- Font: 20px / 600 / 1.3, `--cma-text-primary`
- Rounded top corners only: `border-radius: 8px 8px 0 0`
- Position: immediately above video element, visually part of player card

Control bar bottom corners: `border-radius: 0 0 8px 8px`.

### Seek Bar & Chapter Markers

- Track: `4px` height, `--cma-seek-track` unfilled, `--cma-seek-filled` played portion
- Thumb: `12px × 12px` circle, `--cma-accent`, `border-radius: 50%`
- Thumb hover: expands to `16px × 16px` with `transform: scale(1.33)`, transition `150ms ease`
- **Chapter markers:** Small dots (`6px × 6px`, `border-radius: 50%`) positioned at chapter timestamp offsets on the seek bar track. Color: `--cma-chapter-marker` (reuses `--cma-warning` `#F59E0B`). On hover, dot expands to `10px × 10px` and shows a tooltip with chapter label.
- Chapter tooltip: `--cma-bg-elevated` background, `--cma-text-primary` text, `border-radius: 4px`, font 14px, padding `4px 8px`. Appears above the chapter dot.
- **Hover thumbnail:** `160px × 90px` preview sprite fragment shown above the seek bar when user hovers/drags. Background: `--cma-bg-elevated` behind the sprite image (letterbox fill for non-16:9 sprite tiles). Border: `1px solid var(--cma-border-subtle)`. Border-radius: `4px`.

### Loading State

- Before `canplaythrough`: video area shows `--cma-bg-deepest` background
- media-chrome's `<MediaLoadingIndicator>` renders a spinner (CSS animation, no external asset)
- Spinner color: `--cma-accent`
- Loading text below spinner (14px, `--cma-text-secondary`): "Loading video..."

### Error State

- If video fails to load (`error` event on `<video>`):
  - Player area shows `--cma-bg-surface` card with centered error message
  - Error icon: Unicode `⚠` (U+26A0) or `⨯`, 32px, `--cma-destructive` color
  - Error heading: "Video failed to load" (20px/600, `--cma-text-primary`)
  - Error body: "Please refresh the page to try again." (16px/400, `--cma-text-secondary`)
  - No retry button (refresh is the recovery path; Phase 3 timer state would be invalid on video error anyway)

### Keyboard Focus

- All interactive elements within player (play, seek, mute, volume, seek-buttons) must show `:focus-visible` outline
- Focus ring: `2px solid var(--cma-focus-ring)`, `outline-offset: 2px`
- media-chrome components handle their own focus rings; override only if default browser rings are invisible on dark backgrounds

## Decisions Log

1. **Chapter marker color = warning amber (`#F59E0B`).** Chapter markers flag suspicious moments — amber creates visual association with "attention needed" without the alarm of red (reserved for timer urgency and errors). If L1 reviewers find amber too alarming for benign chapters, tone it down to `#6366F1` (indigo) in Phase 3.

2. **System font stack, not Inter or custom font.** Per project constraint: zero external dependencies. Stack covers Windows (Segoe UI), macOS (SF Pro via -apple-system), Linux (Roboto fallback). Sufficient for a hiring assessment tool.

3. **No `rem` units in Phase 2.** Establish baseline in `px` now. If responsive needs emerge in Phase 6, convert tokens to `rem` in a single pass. `px` is simpler for initial visual QA against design mockups.

4. **`max-width: 960px` for player.** 960px at 16:9 = 540px tall video area. Fits comfortably on 1366px laptop screens (most common moderator resolution) with room for side panels in future phases. 960px also divides cleanly by 8 (the spacing grid).

5. **Single CSS file per domain.** `src/index.css` holds global tokens and reset. `src/components/player/VideoPlayer.css` holds media-chrome overrides. Phase 1 screens get `src/components/LandingScreen.css`, `src/components/GuidelinesScreen.css`, `src/components/AlreadyCompletedScreen.css`. No CSS-in-JS, no CSS modules (keep build zero-config).

6. **No animations in Phase 2.** Static visual system only. GSAP transitions between screens, timer pulse, and button hover micro-interactions land in Phase 6 per ROADMAP.md. This keeps Phase 2 scope tight and avoids drift into motion design.

7. **Button border-radius: 6px (not pill, not square).** Streaming services use rounded-rect buttons — soft enough to feel premium, squared enough to feel professional. 6px matches the input field radius for visual consistency.

8. **Unicode fallback icons, not SVG.** `✓` for success, `⚠` for warnings, `⨯` for errors. media-chrome provides its own control icons. No icon library needed. Phase 4 Lottie animations are decorative (milestones), not functional icons.
