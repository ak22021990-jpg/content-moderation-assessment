## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| TagPanel → RunnerScreen (onSelectionChange) | Tag selections cross from child to parent via callback for verdict snapshot |
| RunnerScreen → useAssessmentStore (commitAnswer) | Answer object built from selections + timer data written to store |
| Candidate DevTools → Zustand (answers[] manipulation) | Client-side state inspectable; server-side validation in Phase 5 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-12 | Tampering | Verdict double-click → duplicate answers | mitigate | Buttons disabled immediately via `submitting` local state. Store commitAnswer appends to array — no dedup needed (Phase 4 scoring processes array sequentially; only last answer per videoIndex is used). |
| T-03-13 | Elevation of Privilege | Candidate modifies currentVideoIndex to skip videos | accept | All client-side state inspectable. Phase 5 server-side validation cross-checks videoIds in answers[] against playlist length. No server-side enforcement before Phase 5. |
| T-03-14 | Information Disclosure | answers[] array visible in DevTools | accept | Answers are candidate's own data — no sensitive info exposed. Phase 4 scoring reads from same array. No PII in answers (identity stored separately). |
| T-03-15 | Denial of Service | VideoPlayerScreen remount on every video advance causes media-chrome state loss | mitigate | key={currentVideoIndex} forces clean remount — intentional. Prevents stale media state (previous video buffering) bleeding into next. Remount overhead is negligible (DOM swap only). |
| T-03-16 | Repudiation | Candidate claims verdict button was clicked accidentally | accept | Verdict is immediate and final (no confirm dialog). This is documented in Guidelines (Phase 1). Assessment design prioritises flow over safety nets. Future: optional confirm dialog gated by client sign-off. |

<risks>
Wave-specific risks:

1. **key={currentVideoIndex} causes flicker.** When video advances, React unmounts and remounts VideoPlayerScreen — the media-chrome player briefly disappears and reappears. Mitigation: the brief visual reset is intentional (signals new video). If undesirable, add CSS transition on RunnerScreen.
2. **TagPanel onSelectionChange stale closure.** If RunnerScreen captures tagSelectionRef in handleVerdict but TagPanel hasn't fired the latest callback yet, verdict uses stale state. Mitigation: useEffect in TagPanel fires synchronously after reducer dispatch (React batches), so ref is updated before the next event loop tick. Verdict click is a separate event loop tick — ref is always current.
3. **App.jsx doesn't handle SCREENS.SCOREBOARD case.** Wave 3 adds the enum but App.jsx wiring happens in Wave 4. If RunnerScreen calls onComplete and App.jsx doesn't handle it, the app may show a blank screen.
