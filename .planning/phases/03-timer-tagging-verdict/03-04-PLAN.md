<risks>
Wave-specific risks:

1. **Integration test mocking complexity.** App.integration.test.jsx now mocks RunnerScreen, which mocks 4 sub-components. If RunnerScreen's real API diverges from mock, integration test stays green while real app breaks. Mitigation: human-verify checkpoint validates real rendering.
2. **onPlaying event fires after seek.** Browser fires playing event on resume after seek. On replaying same video during assessment (e.g., candidate seeks forward), playing refires → store's startTimer guard prevents double-start. Good. BUT if candidate seeks after timer expired (auto-submit fired, nextVideo called), playing fires on the OLD video element's re-seek → timer may start on a video whose index already advanced. Mitigation: key={currentVideoIndex} remounts VideoPlayerScreen — old video element destroyed, can't fire events. Clean.
3. **getState() in App.jsx.** Using `useAssessmentStore.getState().startTimer()` is imperative and does NOT subscribe App.jsx to timer changes. This is intentional. However, if the store hasn't been initialised yet (first render of App), getState() still works — Zustand creates the store on first `create()` call, which is a module-level side effect. Safe.
4. **useAssessmentState tests break.** Changing enterAssessment from SCREENS.ASSESSMENT to SCREENS.RUNNER changes the expected screen value in hook unit tests. Mitigation: update tests to expect SCREENS.RUNNER.
5. **VideoPlayerScreen test breaks.** Adding onPlaying prop is backward-compatible (optional). Existing tests that don't pass onPlaying should still work. The new useEffect adds a listener to videoRef — in happy-dom, videoRef is a div (due to media-chrome mock), so addEventListener('playing') is called on a div. Should not throw — happy-dom supports addEventListener on all elements.
6. **Brand-forbidden strings in scoreboard stub.** "Results will appear here in Phase 4" — check against brand-guardrails.md for any forbidden strings. Generic text is safe.
</risks>
