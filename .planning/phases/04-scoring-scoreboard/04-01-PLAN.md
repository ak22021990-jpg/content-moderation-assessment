<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| performance.now() → store state | Wall-clock-independent monotonic time source; no external input |
| tagSnapshot → buildAnswerSnapshot | tagSnapshot can be null (before first tag selection); defensively guarded with ?? [] |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01-a | Denial of Service | buildAnswerSnapshot — null tagSnapshot | mitigate | Destructure with `state.tagSnapshot?.selectedL1 ?? []` — null tagSnapshot (before first tag interaction) produces safe empty arrays instead of crashing |
| T-04-01-b | Tampering | submittedAt via Date.now() | accept | Client-side timestamp is modifiable by candidate (DevTools). Server-side validation in Phase 5 will cross-check submission timing. No PII in timestamp. |
| T-04-01-c | Information Disclosure | timeSpentMs in answer object | accept | Timing data exposes candidate speed — this is a feature (competency signal), not a leak. Data stays client-side until Phase 5 submission. |
| T-04-01-d | Denial of Service | playlist.json missing currentVideoIndex entry | mitigate | `video?.id ?? ''` fallback prevents crash on malformed playlist. Empty string videoId causes scoring function to mark as "no answer key" rather than crash. |
</threat_model>

<risks>
1. **Existing tests break on answer shape change.** Any test that directly inspects `store.answers[0]` shape will need updating. Mitigation: Step 2 explicitly updates assessmentStore.test.js. Other test suites (RunnerScreen, TagPanel) test UI behavior, not answer shape — they should be unaffected.
2. **Timer auto-submit race condition.** If verdict button click and timer expiry happen in same RAF frame, double-submit possible. Mitigation: setTagSnapshot → commitAnswer clears tagSnapshot; second call would build from null → safe defaults. Low risk — UI already disables buttons during submit (submitting state).
3. **sessionStorage timer restore edge-case.** If timer is restored from sessionStorage after a refresh, startedAt is reset via performance.now() on startTimer(), not the original wall clock. This means timeSpentMs is computed from the restored session's start, not the original. Mitigation: timeSpentMs represents actual time spent in the current browser session (including refresh gap). Acceptable for v1.
