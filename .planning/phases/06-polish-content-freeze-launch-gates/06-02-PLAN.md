Implement the automated brand-leak defense infrastructure: a `.husky/pre-commit` hook that blocks local commits containing forbidden brand strings, and a `.github/workflows/brand-guard.yml` CI workflow that fails PRs on the same patterns. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Developer → git | Pre-commit hook is the boundary between developer intent and git history. A brand string in any staged file is the threat crossing into permanent history. |
| PR → main branch | CI brand-guard.yml is the boundary between a PR and the main branch. A brand string that bypassed the pre-commit hook (e.g., committed from a machine without husky installed) is caught here. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-02a | Information Disclosure | Developer commit | mitigate | .husky/pre-commit blocks commit with exit code 1; grep scans staged diffs for 55 forbidden strings |
| T-06-02b | Information Disclosure | PR merge | mitigate | CI brand-guard.yml fails PR check; `git grep` scans all tracked files (catches commits from machines without husky) |
| T-06-02c | Tampering | Pre-commit bypass | mitigate | Developer can `git commit --no-verify` to skip hook — CI is the defense-in-depth gate. Documented in brand-guardrails.md that --no-verify circumvents local protection. |
| T-06-02d | Repudiation | Hook removal | accept | A malicious developer can delete .husky/pre-commit. Trust model assumes good-faith developers; CI is the audit trail. No technical mitigation for intentional bypass. |
| T-06-SC | Tampering | npm installs | mitigate | slopcheck audit in 06-RESEARCH.md — zero new packages. Husky v9 already in devDependencies and verified. |
